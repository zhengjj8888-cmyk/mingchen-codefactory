# Mingchen CodeFactory

> 在写代码前，先看懂这套代码是怎样运转的。

Mingchen CodeFactory 是一项面向陌生项目、Bug 排查、重构和代码审查的 Codex Skill。它要求 AI 先通过代码图谱理解项目结构、调用关系和影响范围，再提出修改建议或开始改代码。

它的价值不在于“搜到一个文件”，而在于回答更关键的问题：**从哪里进来、会影响到哪里、改完应该验证什么。**

## 30 秒理解它

当你要改一个接口、删除一个函数、排查一条请求，或接手一个陌生项目时，最容易犯的错误是：看到一个同名文件，就以为找到了答案。

这个 Skill 会要求 AI 先建立结构化证据链：

1. 找到真正的入口、符号或功能节点；
2. 看它的调用者、被调用者和关联关系；
3. 区分直接影响与需要人工确认的潜在影响；
4. 找出要同步检查的测试、配置、路由或服务边界；
5. 在修改后重新验证关键路径。

## 它适合解决什么问题

| 你的问题 | 它会帮你确认什么 |
| --- | --- |
| “这个功能到底在哪里实现？” | 实际文件、函数、类或路由入口，而不是只给关键词命中 |
| “改这个接口会影响什么？” | 直接调用、上下游依赖、配置、测试和跨服务风险 |
| “这个函数能删吗？” | 调用者、注册项、路由、配置和测试是否仍在使用 |
| “这条请求怎么走到这里的？” | 从入口到处理逻辑的关键调用链 |
| “接手一个项目，先看什么？” | 包边界、入口点、服务连接、热点和风险区域 |
| “这个 Bug 应该从哪里查？” | 最有可能的入口、状态变化和关联模块 |

## 它和普通文本搜索有什么不同

普通搜索擅长找字符串；CodeFactory 更关注代码里的关系：

- 谁调用了它；
- 它又调用了谁；
- 它挂在哪个路由、服务或事件流程上；
- 改动会穿过哪些模块边界；
- 哪些测试、配置和类型需要一起看。

所以它特别适合“看起来改一行，实际会牵动一串”的任务。

## 使用前需要什么

这个仓库发布的是 **Codex Skill**，不包含代码图谱 MCP 服务本身。

使用前，需要为目标项目准备兼容的 `mingchen-codefactory` MCP 服务，并确保项目已建立或可建立索引。没有服务或索引时，Skill 不会假装有图谱结果；它会明确把结论降级为普通文件搜索层面的判断。

## 安装

```bash
git clone https://github.com/zhengjj8888-cmyk/mingchen-codefactory.git
cp -R mingchen-codefactory ~/.codex/skills/
```

完成 MCP 服务配置并为目标项目建立索引后，即可使用。

## 第一次使用：按这 4 步来

### 第 1 步：准备代码图谱服务

本 Skill 需要兼容的 `mingchen-codefactory` MCP 服务。服务应至少能提供项目索引、结构搜索、调用链追踪、架构摘要和代码片段定位等能力。

如果你还没有这项服务，可以先安装 Skill；但它无法凭空生成项目图谱。

### 第 2 步：为项目建立索引

在准备分析的项目上完成首次索引。索引更新后，Skill 才能基于真实的符号、文件、调用关系和架构边界给出结论。

### 第 3 步：安装并重启 Codex

按上面的命令把仓库复制到 `~/.codex/skills/`，然后重新打开 Codex，让它识别 `$mingchen-codefactory`。

### 第 4 步：先分析，再修改

第一次建议直接这样说：

```text
使用 $mingchen-codefactory。
我刚接手这个项目。请先告诉我项目入口、主要模块、关键服务和测试位置。
先给结构图和风险点，不要改代码。
```

之后再告诉它你要修改的功能、接口或 Bug。涉及删除、重构和跨服务改动时，先要求“影响分析”，确认后再让它实现。

## 怎么提需求

下面这些提示可以直接复制：

### 1. 找到真实实现位置

```text
使用 $mingchen-codefactory。
我想知道“订单取消”到底在哪些地方实现。
先给我入口、核心处理函数、相关状态变更和对应测试，不要直接改代码。
```

### 2. 改接口前做影响分析

```text
使用 $mingchen-codefactory。
我准备给 createOrder 接口新增一个字段。
请先分析会受影响的路由、调用方、类型、校验、数据库映射和测试，再给我修改方案。
```

### 3. 判断能否删除

```text
使用 $mingchen-codefactory。
这个 legacySync 函数看起来已经不用了。
请检查调用者、事件注册、配置引用和测试依赖，告诉我能不能删除，以及删除后要验证什么。
```

### 4. 排查请求链路

```text
使用 $mingchen-codefactory。
用户提交退款申请后，状态没有变成 processing。
请从接口入口开始追踪到状态写入，找出关键节点和最可能的断点。先分析，不要直接修改。
```

## 你会拿到什么

每次分析应尽量给出以下内容：

1. **定位结果**：具体符号、文件和代码位置；
2. **调用链**：请求或事件从哪里进入，关键经过什么节点；
3. **直接影响**：必须同步改动的类型、配置、路由、测试或模块；
4. **潜在影响**：权限、缓存、异步、跨服务或历史兼容问题；
5. **验证建议**：改完后需要跑的测试、查询和人工检查。

## 使用边界

- 它帮助你看懂系统，不替代代码审查、测试和上线验证。
- 索引可能过期；高风险修改前应刷新索引或重新建立证据。
- 图谱结果是决策依据，不是“可以直接删改”的授权。
- 不要把代码库中的密钥、客户数据或生产日志放进公开问题描述。

## 常见问题

**没有 MCP 服务还能用吗？**  
可以把它当作一套分析习惯参考，但它无法提供真正的代码图谱、调用链和影响范围证据。要获得完整能力，需要先接入兼容服务。

**它会直接改代码吗？**  
可以，但建议先让它完成定位和影响分析。对重构、删除和接口改动来说，这一步往往比开始改更重要。

**适合什么规模的项目？**  
小项目可用于快速理解入口和依赖；项目越大、模块越多、服务边界越复杂，它的结构化分析价值越明显。
